## How the schema was derived

Each JSON Schema was generated by reading the corresponding prose section and translating:

1. **Required fields** → JSON Schema `required` array
2. **Optional fields** → listed in `properties` but not in `required`
3. **Opaque token fields** (§3.1.1) → `type: string` with descriptive `description` (no pattern, since they are opaque by design)
4. **Enumerated values** (e.g., `role`, `scimType`) → JSON Schema `enum`
5. **Format constraints** (RFC 3339 timestamps) → JSON Schema `format: date-time`
6. **Array constraints** (e.g., `decisionOptions` must be non-empty) → `minItems: 1`
7. **Schema URIs** → `const` in array items or `$ref` targets
8. **Cross-references** (e.g., Decision referencing a ScopeItem's decisionOptions) → documented in `description` since the spec does not define a formal extension mechanism for cross-resource validation

## Relationship to the prose spec

**The prose spec is authoritative.** If a JSON Schema and the prose spec disagree, the prose spec wins. The schemas are a convenience for implementation and validation — they are not a replacement for reading the spec.

## Open questions carried forward

The following items from the prose spec's "Open Questions" (§12) are reflected in the schemas:

- **Q8 (BulkResponse):** The schema for the batch response is not yet defined. The OpenAPI spec notes SCIM's `BulkResponse` as the likely path. See `docs/schema-findings.md` (F1).
- **Q9 (Baseline signing algorithm):** The signing algorithm list in `capabilities.schema.json` is unconstrained. If Q9 is resolved to mandate ES256, the schema would need a `minItems` constraint or a default.

See `docs/schema-findings.md` for all findings discovered during schema formalization.
